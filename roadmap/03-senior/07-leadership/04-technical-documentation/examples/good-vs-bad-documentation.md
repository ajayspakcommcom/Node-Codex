# Good Vs Bad Documentation

## Bad

- long prose with no clear action path during incidents
- architecture docs with no ownership or update expectation
- onboarding docs that only list commands without explaining system context
- runbooks that describe alerts but not what to do next

## Good

- concise docs with clear purpose and ownership
- documents that explain constraints, failure modes, and decisions
- runbooks with trigger, steps, escalation, and rollback guidance
- onboarding docs that connect code, systems, and operations
