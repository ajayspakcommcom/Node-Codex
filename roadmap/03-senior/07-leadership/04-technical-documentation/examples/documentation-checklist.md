# Documentation Checklist

- Does the document explain the system or decision clearly enough for someone new to the area?
- Does it describe important constraints, tradeoffs, and failure modes?
- Is ownership explicit?
- Is the document useful during onboarding, change review, or incident response?
- Are links to dashboards, runbooks, contracts, or code paths included where relevant?
- Is the document short enough to scan but complete enough to act on?
- Is there a clear update expectation to prevent drift?
