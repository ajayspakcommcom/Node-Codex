# Write RFCs

## Purpose

This topic is about writing technical proposals that clarify decisions, tradeoffs, migration paths, and risks before implementation begins, especially when multiple teams or stakeholders are involved.

## Enterprise-Level Pointers

- what an RFC is and when to write one
- problem framing and decision scope
- alternatives and tradeoff analysis
- migration strategy and rollout planning
- compatibility and risk assessment
- decision records and unresolved questions
- stakeholder review and alignment process
- keeping RFCs actionable instead of academic
- linking RFCs to implementation and follow-up
- common production mistakes
- maintainability rules
- interview-style questions
- practice exercises

## What Expert Teams Optimize For

- reducing rework by aligning on decisions early
- making tradeoffs explicit and reviewable
- giving future maintainers clear context for why a change happened
- separating strategic direction from implementation churn

## Common Production Mistakes

- writing RFCs that only restate the solution without the problem
- omitting alternatives and tradeoffs
- treating RFC approval as final truth while implementation diverges
- writing documents too vague to guide real engineering work

## Maintainability Rules

- define the problem, decision, and non-goals clearly
- include risks, alternatives, and migration steps explicitly
- keep RFCs linked to implementation outcomes
- update status and decision records as the work evolves

## Interview Questions

- When is an RFC justified instead of a quick design note?
- Why are alternatives and non-goals critical in an RFC?
- How do you keep RFCs from becoming stale documents?

## Practice Exercises

- Draft an RFC outline for adopting a new service platform component.
- Compare two competing designs and record the tradeoffs.
- Create a migration and rollback section for an architecture RFC.
